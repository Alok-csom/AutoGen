In this lab you'll build a working multi-agent AI system from scratch вҖ” from raw audio files to a live demo with three user roles. We'll use AutoGen Studio's drag-and-drop interface, Azure AI Foundry (GPT-4.1), and ChromaDB. No security backend needed.

Overview & Learning Objectives
Read this before we start

In this lab you'll get hands-on experience building a multi-agent AI system вҖ” not just the theory. By the end of the session you'll have a working system that demonstrates role-based access control using nothing more than conversational prompts and retrieval filtering. No authentication server, no backend вҖ” just agents doing exactly what they're designed to do.

The scenario: a hospital has audio recordings of doctor-patient conversations. Different users вҖ” doctors, patients, and pharmacists вҖ” need to query those records, but each should only see what they're authorized to see. We'll build an agent system that enforces this automatically.

Understand before you build

Before we write any code, let's look at how the whole system fits together. There are two distinct phases: an offline setup phase (Steps 1вҖ“5, which you'll complete before class) and a live conversation phase (Steps 6вҖ“10, which we'll build together). Understanding this split matters because it determines where each component lives and why.

рҹ’Ў Why metadata filtering matters
The where filter in ChromaDB is the real access boundary вҖ” not the system prompt. A pharmacist's query physically only retrieves records tagged section=prescription. The LLM cannot leak diagnosis data because it never receives it. This distinction вҖ” prompt-based restriction versus data-layer filtering вҖ” is one of the key concepts we'll come back to when we run the pharmacist scenario in the demo.

Write the Transcription Tool
вҸұ ~8 minutes

Save this as tools/transcription_tool.py. This is the first of your three tool files. It uses Whisper to convert audio to text, then calls GPT-4.1 to extract three structured fields from the raw transcript: summary, diagnosis, and prescription. Notice how the LLM is doing structured extraction here вҖ” we'll come back to why this matters when we get to the ChromaDB filtering in Step 4.

tools/transcription_tool.py
import whisper, json, os, openai
from dotenv import load_dotenv

load_dotenv()

def extract_sections(transcript: str, patient_id: str) -> dict:
    """Use GPT-4.1 to split a raw transcript into summary, diagnosis, prescription."""
    client = openai.AzureOpenAI(
        api_key=os.environ["AZURE_OPENAI_API_KEY"],
        azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
        api_version=os.environ["AZURE_API_VERSION"]
    )
    response = client.chat.completions.create(
        model=os.environ["AZURE_DEPLOYMENT_NAME"],
        messages=[
            {
                "role": "system",
                "content": (
                    "Extract three fields from this doctor-patient transcript as JSON. "
                    "Return ONLY valid JSON with keys: summary, diagnosis, prescription. "
                    "If a field is not present, return an empty string for that key."
                )
            },
            {"role": "user", "content": transcript}
        ],
        response_format={"type": "json_object"}
    )
    data = json.loads(response.choices[0].message.content)
    data["patient_id"] = patient_id
    return data


def transcribe_all(audio_dir: str = 'audio_files') -> str:
    """Transcribe all audio files and extract structured sections."""
    model = whisper.load_model("base")
    records = []

    for fname in sorted(os.listdir(audio_dir)):
        if not fname.endswith(('.mp3', '.wav')):
            continue
        patient_id = fname.split('_')[0]   # e.g. P001_visit.mp3 вҶ’ P001
        fpath = os.path.join(audio_dir, fname)

        print(f"Transcribing {fname}...")
        result = model.transcribe(fpath)
        structured = extract_sections(result["text"], patient_id)
        records.append(structured)

    # Load existing records and only add new patients вҖ” never overwrite
    existing = []
    if os.path.exists("transcripts.json"):
        with open("transcripts.json") as f:
            existing = json.load(f)

    existing_ids = {r["patient_id"] for r in existing}
    new_records = [r for r in records if r["patient_id"] not in existing_ids]

    with open("transcripts.json", "w") as f:
        json.dump(existing + new_records, f, indent=2)

    return f"Transcribed {len(new_records)} new file(s). transcripts.json now has {len(existing + new_records)} patient(s)."


if __name__ == "__main__":
    print(transcribe_all())

Write the Vector Store Tool
вҸұ ~8 minutes

Save as tools/vector_store.py. This file is the heart of the whole system. Pay close attention to the where filter in each query function вҖ” that single parameter is what makes role-based access work. Run the test and look at what the three different query outputs tell you.

tools/vector_store.py
import chromadb, json
from sentence_transformers import SentenceTransformer

embedder = SentenceTransformer("all-MiniLM-L6-v2")
chroma_client = chromadb.Client()   # In-memory for demo
collection = chroma_client.get_or_create_collection("medical_records")


def ingest_transcripts(json_path: str = 'transcripts.json') -> str:
    """Load transcripts.json and ingest all sections into ChromaDB."""
    with open(json_path) as f:
        records = json.load(f)

    count = 0
    for rec in records:
        pid = rec["patient_id"]
        for section in ["summary", "diagnosis", "prescription"]:
            text = rec.get(section, '').strip()
            if not text: continue
            embedding = embedder.encode(text).tolist()
            collection.add(
                documents=[text],
                embeddings=[embedding],
                metadatas=[{"patient_id": pid, "section": section}],
                ids=[f"{pid}_{section}"]
            )
            count += 1
    return f"Ingested {count} sections from {len(records)} patient records."


def query_all(question: str) -> str:
    """Doctor access: no filter вҖ” returns all matching records."""
    n   = collection.count()  # dynamically retrieve all available sections
    emb = embedder.encode(question).tolist()
    results = collection.query(query_embeddings=[emb], n_results=n)
    return _format(results)


def query_patient(question: str, patient_id: str) -> str:
    """Patient access: filter by patient_id only."""
    n   = collection.count()  # dynamically retrieve all available sections
    emb = embedder.encode(question).tolist()
    results = collection.query(
        query_embeddings=[emb],
        where={"patient_id": patient_id},  # вҶҗ access boundary
        n_results=n
    )
    return _format(results)


def query_prescriptions(question: str) -> str:
    """Pharmacist access: prescription sections only."""
    n   = collection.count()  # dynamically retrieve all available sections
    emb = embedder.encode(question).tolist()
    results = collection.query(
        query_embeddings=[emb],
        where={"section": "prescription"},  # вҶҗ access boundary
        n_results=n
    )
    return _format(results)


def _format(results: dict) -> str:
    docs  = results.get('documents', [[]])[0]
    metas = results.get('metadatas', [[]])[0]
    if not docs: return "No relevant records found."
    lines = []
    for doc, meta in zip(docs, metas):
        pid = meta.get("patient_id", "?")
        sec = meta.get("section", "?")
        lines.append(f"[Patient {pid} | {sec.upper()}] {doc}")
    return "\n\n".join(lines)


if __name__ == "__main__":
    print(ingest_transcripts())
    print(query_all("What prescriptions were given?"))
    print(query_patient("What is my diagnosis?", "P001"))
    print(query_prescriptions("What medications were prescribed?"))

рҹ”‘ Key concept вҖ” where filtering is your access control
The where={"section": "prescription"} filter runs inside ChromaDB before any data reaches the LLM. This is not prompt-based security вҖ” it's database-level filtering. A pharmacist's agent physically cannot retrieve diagnosis or summary text, regardless of what they ask. Here's a question worth sitting with: what's the difference between telling an LLM not to reveal something versus ensuring it never receives that data in the first place?

New Architecture вҖ” Two Agents, One Team
вҸұ Read this before building anything in AutoGen Studio

After extensive testing we discovered that having both a team selector node and an OrchestratorAgent trying to route conversations creates conflicts. The selector LLM and the OrchestratorAgent's system prompt fight each other for control.

The new approach is simpler and more reliable:

рҹ”‘ New architecture вҖ” two agents only
Agent 1 вҖ” User (UserProxyAgent): represents you in the conversation. Gets selected when the system needs human input.

Agent 2 вҖ” MedicalAgent (AssistantAgent): handles everything вҖ” loading data, asking for role, routing to the right tool based on role. Has ALL tools attached.

The team selector node does one simple job: after MedicalAgent speaks вҶ’ select User. After User speaks вҶ’ select MedicalAgent. That's it. No role logic in the selector at all вҖ” MedicalAgent handles all the conversation logic through its system prompt.

Why this works better

Instead of the selector trying to interpret conversation content and pick between five specialist agents, MedicalAgent reads the conversation itself and calls the right tool directly. The selector just alternates between MedicalAgent and User вҖ” a pattern it can follow reliably every time.

Old approachNew approach
5 specialist agents + OrchestratorAgent + User = 7 nodesMedicalAgent + User = 2 nodes
Selector tries to route between 6 AI agentsSelector just alternates User вҶ” MedicalAgent
OrchestratorAgent conflicts with selectorNo conflict вҖ” one agent, one selector rule
Role logic split between selector and system promptsAll logic in MedicalAgent system prompt

import sys, json, os
# вҡ пёҸ Replace the path below with YOUR actual tools folder path (use forward slashes)
sys.path.insert(0, '/path/to/your/medical-demo/tools')  # e.g. 'C:/Users/yourname/autogen-demo/medical-demo/tools'
import whisper
from transcription_tool import extract_sections
from vector_store import ingest_transcripts, collection

def process_audio_files(audio_dir: str = 'audio_files') -> str:
    """Transcribe only NEW audio files not already in ChromaDB. Safe to call multiple times."""
    existing = collection.get()
    indexed = set(m["patient_id"] for m in existing["metadatas"])
    model = whisper.load_model("base")
    new_records = []
    for fname in sorted(os.listdir(audio_dir)):
        if not fname.endswith(('.mp3', '.wav')): continue
        patient_id = fname.split('_')[0]
        if patient_id in indexed: continue
        result = model.transcribe(os.path.join(audio_dir, fname))
        new_records.append(extract_sections(result["text"], patient_id))
    if not new_records: return "No new files found."
    existing_data = json.load(open('transcripts.json'))
    with open('transcripts.json', 'w') as f:
        json.dump(existing_data + new_records, f, indent=2)
    ingest_transcripts('transcripts.json')
    return f"вң“ Transcribed and indexed {len(new_records)} new patient(s): {[r['patient_id'] for r in new_records]}"

Azure endpoint format вҖ” fix this if you get a 404

рҹ”‘ The most common Azure 404 cause
The Azure AI Foundry "View Code" page shows an endpoint like https://your-resource.openai.azure.com/openai/v1. Do not paste this into your .env file as-is. The openai.AzureOpenAI client constructs the full URL by appending its own path to whatever endpoint you give it. If the endpoint already has /openai/v1 baked in, the final URL ends up doubled вҖ” something like .../openai/v1/openai/deployments/... вҖ” which gives a 404.

Always strip the path suffix. Your endpoint in .env should be the base URL only:
.env вҖ” correct format
# вң… Correct вҖ” base URL only, trailing slash, no path suffix
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/

# вқҢ Wrong вҖ” has /openai/v1 suffix copied from Foundry View Code page
# AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/openai/v1
рҹ’Ў Also comment out api_version
The newer Azure AI Foundry endpoints don't require an api_version parameter. If your .env doesn't include AZURE_API_VERSION, comment it out in your Python code rather than trying different version strings вҖ” changing API versions introduces dependency conflicts with other packages. In transcription_tool.py and any other file that creates an AzureOpenAI client, comment the line out:
transcription_tool.py вҖ” comment out api_version
client = openai.AzureOpenAI(
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    # api_version not required for Azure AI Foundry endpoints
    # api_version=os.environ["AZURE_API_VERSION"]
)

ffmpeg not found (Windows) вҖ” fix this first

If you see a FileNotFoundError: [WinError 2] The system cannot find the file specified error when running the transcription tool, Whisper can't find ffmpeg вҖ” it needs it to decode .mp3 files. Install it with one of these:

Option A вҖ” winget (recommended, built into Windows 10/11)

PowerShell (VS Code) вҖ” run as Administrator
winget install ffmpeg

After installing, close and reopen your PowerShell terminal so it picks up the new PATH. Then verify:

PowerShell (VS Code)
ffmpeg -version
# Should print version info вҖ” if it does, you're good to go

Option B вҖ” Manual install (no package manager needed)

  1. Go to https://ffmpeg.org/download.html вҶ’ Windows вҶ’ Windows builds by BtbN
  2. Download ffmpeg-master-latest-win64-gpl.zip
  3. Extract it, rename the folder to ffmpeg, move it to C:\ffmpeg
  4. Search "Edit environment variables" вҶ’ click Path вҶ’ New вҶ’ type C:\ffmpeg\bin
  5. Click OK, then close and reopen PowerShell

Option C вҖ” Use .wav files instead (avoids ffmpeg entirely)

Whisper can read .wav files without needing ffmpeg. If you generate your TTS audio as .wav instead of .mp3, the error goes away completely. Most TTS tools support both formats вҖ” just save as P001_visit.wav etc. and update the file extension check in the tool code if needed.
