Building a Role-Based Medical Q&A System with AutoGen Studio

Prof. Alok Gupta  ·  Curtis L. Carlson Chair in Information Management

In this lab you'll build a working multi-agent AI system from scratch — from raw audio files to a live demo with three user roles. We'll use AutoGen Studio's drag-and-drop interface, Azure AI Foundry (GPT-4.1), and ChromaDB. No security backend needed.

AutoGen Studio v0.4 Azure AI Foundry ChromaDB ~75 min total 5 agents · 3 roles

Overview & Learning Objectives
Read this before we start

In this lab you'll get hands-on experience building a multi-agent AI system — not just the theory. By the end of the session you'll have a working system that demonstrates role-based access control using nothing more than conversational prompts and retrieval filtering. No authentication server, no backend — just agents doing exactly what they're designed to do.

The scenario: a hospital has audio recordings of doctor-patient conversations. Different users — doctors, patients, and pharmacists — need to query those records, but each should only see what they're authorized to see. We'll build an agent system that enforces this automatically.

System Architecture
Understand before you build

Before we write any code, let's look at how the whole system fits together. There are two distinct phases: an offline setup phase (Steps 1–5, which you'll complete before class) and a live conversation phase (Steps 6–10, which we'll build together). Understanding this split matters because it determines where each component lives and why.

Input
🎙️ User sends audio folder path in Playground chat
↓
Layer 1b
LabAssistantAgent → receives upload message → calls process_audio_files tool
↓
Layer 2
TranscriptionAgent → Whisper → GPT-4.1 section extraction → structured JSON
↓
Layer 3
VectorStoreAgent → embed each section → ChromaDB with patient_id + section metadata
↓
Layer 4
OrchestratorAgent → "Are you a doctor, patient, or pharmacist?" → routes to specialist
↓
Layer 5
Doctor (all) · Patient (own only) · Pharmacist (prescriptions only)
↓
Output
Scoped answer returned to user — filtered at retrieval, not by prompt alone
💡 Why metadata filtering matters
The where filter in ChromaDB is the real access boundary — not the system prompt. A pharmacist's query physically only retrieves records tagged section=prescription. The LLM cannot leak diagnosis data because it never receives it. This distinction — prompt-based restriction versus data-layer filtering — is one of the key concepts we'll come back to when we run the pharmacist scenario in the demo.

The interactive chat interface for testing your team in real time. Features include:

Write the Transcription Tool
⏱ ~8 minutes

Save this as tools/transcription_tool.py. This is the first of your three tool files. It uses Whisper to convert audio to text, then calls GPT-4.1 to extract three structured fields from the raw transcript: summary, diagnosis, and prescription. Notice how the LLM is doing structured extraction here — we'll come back to why this matters when we get to the ChromaDB filtering in Step 4.

tools/transcription_tool.py
import whisper, json, os, openai
from dotenv import load_dotenv

load_dotenv()

def extract_sections(transcript: str, patient_id: str) -> dict:
    """Use GPT-4.1 to split a raw transcript into summary, diagnosis, prescription."""
    client = openai.AzureOpenAI(
        api_key=os.environ["AZURE_OPENAI_API_KEY"],
        azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
        api_version=os.environ["AZURE_API_VERSION"]
    )
    response = client.chat.completions.create(
        model=os.environ["AZURE_DEPLOYMENT_NAME"],
        messages=[
            {
                "role": "system",
                "content": (
                    "Extract three fields from this doctor-patient transcript as JSON. "
                    "Return ONLY valid JSON with keys: summary, diagnosis, prescription. "
                    "If a field is not present, return an empty string for that key."
                )
            },
            {"role": "user", "content": transcript}
        ],
        response_format={"type": "json_object"}
    )
    data = json.loads(response.choices[0].message.content)
    data["patient_id"] = patient_id
    return data


def transcribe_all(audio_dir: str = 'audio_files') -> str:
    """Transcribe all audio files and extract structured sections."""
    model = whisper.load_model("base")
    records = []

    for fname in sorted(os.listdir(audio_dir)):
        if not fname.endswith(('.mp3', '.wav')):
            continue
        patient_id = fname.split('_')[0]   # e.g. P001_visit.mp3 → P001
        fpath = os.path.join(audio_dir, fname)

        print(f"Transcribing {fname}...")
        result = model.transcribe(fpath)
        structured = extract_sections(result["text"], patient_id)
        records.append(structured)

    with open("transcripts.json", "w") as f:
        json.dump(records, f, indent=2)

    return f"Transcribed {len(records)} files. Saved to transcripts.json."


if __name__ == "__main__":
    print(transcribe_all())

Write the Vector Store Tool
⏱ ~8 minutes

Save as tools/vector_store.py. This file is the heart of the whole system. Pay close attention to the where filter in each query function — that single parameter is what makes role-based access work. Run the test and look at what the three different query outputs tell you.

tools/vector_store.py
import chromadb, json
from sentence_transformers import SentenceTransformer

embedder = SentenceTransformer("all-MiniLM-L6-v2")
chroma_client = chromadb.Client()   # In-memory for demo
collection = chroma_client.get_or_create_collection("medical_records")


def ingest_transcripts(json_path: str = 'transcripts.json') -> str:
    """Load transcripts.json and ingest all sections into ChromaDB."""
    with open(json_path) as f:
        records = json.load(f)

    count = 0
    for rec in records:
        pid = rec["patient_id"]
        for section in ["summary", "diagnosis", "prescription"]:
            text = rec.get(section, '').strip()
            if not text: continue
            embedding = embedder.encode(text).tolist()
            collection.add(
                documents=[text],
                embeddings=[embedding],
                metadatas=[{"patient_id": pid, "section": section}],
                ids=[f"{pid}_{section}"]
            )
            count += 1
    return f"Ingested {count} sections from {len(records)} patient records."


def query_all(question: str, n: int = 5) -> str:
    """Doctor access: no filter — returns all matching records."""
    emb = embedder.encode(question).tolist()
    results = collection.query(query_embeddings=[emb], n_results=n)
    return _format(results)


def query_patient(question: str, patient_id: str, n: int = 5) -> str:
    """Patient access: filter by patient_id only."""
    emb = embedder.encode(question).tolist()
    results = collection.query(
        query_embeddings=[emb],
        where={"patient_id": patient_id},  # ← access boundary
        n_results=n
    )
    return _format(results)


def query_prescriptions(question: str, n: int = 5) -> str:
    """Pharmacist access: prescription sections only."""
    emb = embedder.encode(question).tolist()
    results = collection.query(
        query_embeddings=[emb],
        where={"section": "prescription"},  # ← access boundary
        n_results=n
    )
    return _format(results)


def _format(results: dict) -> str:
    docs  = results.get('documents', [[]])[0]
    metas = results.get('metadatas', [[]])[0]
    if not docs: return "No relevant records found."
    lines = []
    for doc, meta in zip(docs, metas):
        pid = meta.get("patient_id", "?")
        sec = meta.get("section", "?")
        lines.append(f"[Patient {pid} | {sec.upper()}] {doc}")
    return "\n\n".join(lines)


if __name__ == "__main__":
    print(ingest_transcripts())
    print(query_all("What prescriptions were given?"))
    print(query_patient("What is my diagnosis?", "P001"))
    print(query_prescriptions("What medications were prescribed?"))
🔑 Key concept — where filtering is your access control
The where={"section": "prescription"} filter runs inside ChromaDB before any data reaches the LLM. This is not prompt-based security — it's database-level filtering. A pharmacist's agent physically cannot retrieve diagnosis or summary text, regardless of what they ask. Here's a question worth sitting with: what's the difference between telling an LLM not to reveal something versus ensuring it never receives that data in the first place?

Tool 4 — process_audio_files (for LabAssistantAgent)

Gallery → Tools → + New Tool
import sys, json, whisper, os
sys.path.insert(0, '/path/to/your/medical-demo/tools')
from transcription_tool import extract_sections
from vector_store import ingest_transcripts, collection

def process_audio_files(audio_dir: str = 'audio_files') -> str:
    """Transcribe only NEW audio files not already in ChromaDB. Safe to call multiple times."""
    existing = collection.get()
    indexed = set(m["patient_id"] for m in existing["metadatas"])
    model = whisper.load_model("base")
    new_records = []
    for fname in sorted(os.listdir(audio_dir)):
        if not fname.endswith(('.mp3', '.wav')): continue
        patient_id = fname.split('_')[0]
        if patient_id in indexed: continue
        result = model.transcribe(os.path.join(audio_dir, fname))
        new_records.append(extract_sections(result["text"], patient_id))
    if not new_records: return "No new files found."
    existing_data = json.load(open('transcripts.json'))
    with open('transcripts.json', 'w') as f:
        json.dump(existing_data + new_records, f, indent=2)
    ingest_transcripts('transcripts.json')
    return f"✓ Transcribed and indexed {len(new_records)} new patient(s): {[r['patient_id'] for r in new_records]}"

🎯 What to say during the pharmacist refusal
Notice what just happened — the pharmacist was refused access to the diagnosis. This isn't a prompt instruction saying 'don't tell pharmacists.' The retrieval tool filtered at the database level, so the LLM never received that data in the first place. There was nothing to share. This is the core design principle to take away: in a trustworthy AI system, access control happens at the data layer, not the language layer.

Troubleshooting
Common issues & fixes
SymptomCauseFix
"No records found" from any agent startup.py was not run, or run in a different terminal Run python startup.py then autogenstudio ui in the same terminal
Azure 401 or 404 error Wrong endpoint, key, or deployment name Base URL must end with .openai.azure.com/ (trailing slash). Deployment name must match Foundry exactly.
A specialist agent replies before role is verified Selector prompt not routing OrchestratorAgent first Add to selector prompt: "If this is the first message, ALWAYS select OrchestratorAgent."
PharmacistAgent reveals diagnosis data Tool code missing the where filter Verify query_prescriptions_only uses where={"section": "prescription"}
Whisper is very slow Running large model on CPU Use use_mock=True in startup.py, or switch to whisper.load_model("tiny")
Tools not found by AutoGen Studio sys.path not pointing to your tools folder Use the full absolute path in sys.path.insert(), not a relative one
