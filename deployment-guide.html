From Prototype to Production: Deploying Your AutoGen System

You've built a working multi-agent prototype in AutoGen Studio. This guide walks you through the steps to take it from a local demo to a real deployed application — with proper authentication, persistent storage, an API layer, and cloud hosting.

Prototype vs. Production — What Actually Changes
Read this before diving into the phases

AutoGen Studio is an excellent prototyping tool — it lets you build, test, and iterate on agent systems quickly. But it is explicitly not designed for production deployment. Here is what changes when you move from the Studio prototype to a real application.

🔑 The core architectural shift
In the prototype, the LLM enforces access control through its system prompt. In production, access control moves upstream of the LLM — your API layer verifies the user's token, extracts their role, and passes only that role to the agent. The LLM never decides who you are. It only decides what to do with a role it has already been told.
✦
The Gap to Bridge
Specifically what the lab prototype is missing

Our medical demo has three specific gaps that must be addressed before it could be used with real patient data:

🚨
Gap 1 — Conversational authentication
Any user can claim any role. Nothing verifies identity. A patient could type "I am a doctor" and get full access.
⚠️
Gap 2 — In-memory vector store
ChromaDB resets when the process restarts. All transcribed records are lost between sessions.
⚠️
Gap 3 — No API surface
The system only works through AutoGen Studio's UI. No other application — web front end, mobile app, EHR system — can call it.

The three phases below address each gap in order. Work through them sequentially — authentication must come before the API layer, and the API layer before deployment.

1
Export Your Team from AutoGen Studio
Phase 1 — Getting the code out of the GUI

AutoGen Studio stores your team configuration as JSON in a local SQLite database. The Deployment tab gives you two export options — both are useful for different purposes.

Option A — Export as Python code (recommended starting point)

  1. In AutoGen Studio, click the Deployment tab in the left sidebar
  2. Select MedicalRecordsTeam from the team dropdown
  3. Click Export to Python
  4. Save the generated team.py file into your medical-demo/ folder

The exported file will look roughly like this — a fully runnable Python script that recreates your team programmatically:

team.py — auto-generated by AutoGen Studio
import asyncio
from autogen_agentchat.agents import AssistantAgent
from autogen_agentchat.teams import SelectorGroupChat
from autogen_agentchat.conditions import MaxMessageTermination
from autogen_ext.models.openai import AzureOpenAIChatCompletionClient
import os
from dotenv import load_dotenv

load_dotenv()

model_client = AzureOpenAIChatCompletionClient(
    azure_deployment=os.environ["AZURE_DEPLOYMENT_NAME"],
    api_version=os.environ["AZURE_API_VERSION"],
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
)

lab_assistant = AssistantAgent(
    name="LabAssistantAgent",
    model_client=model_client,
    tools=[process_mock_files, process_audio_files],
    system_message="..."
)
orchestrator = AssistantAgent(name="OrchestratorAgent", model_client=model_client, system_message="...")
doctor_agent  = AssistantAgent(name="DoctorAgent",      model_client=model_client, tools=[query_all_records],         system_message="...")
patient_agent = AssistantAgent(name="PatientAgent",     model_client=model_client, tools=[query_patient_records],     system_message="...")
pharma_agent  = AssistantAgent(name="PharmacistAgent",  model_client=model_client, tools=[query_prescriptions_only],  system_message="...")

termination = MaxMessageTermination(max_messages=20)

    participants=[lab_assistant, orchestrator, doctor_agent, patient_agent, pharma_agent],
    model_client=model_client,
    termination_condition=termination,
    selector_prompt="..."

    result = await team.run(task=message)
    return result

Option B — Export as JSON (for configuration management)

Click Export to JSON to get the raw team configuration. This is useful for version-controlling your agent definitions in Git without the Python boilerplate:

terminal — serve directly from JSON (quick test)
# Serve the exported team as a REST endpoint directly
autogenstudio serve --team team_config.json --port 8084
💡 Which to use
Use the Python export as your starting point for production — it gives you code you can modify, version control, and integrate with FastAPI. Use the JSON export for quick tests or if you want to commit agent definitions separately from application code.
2
Replace Conversational Auth with Real Authentication
Phase 2 — The most critical change for a medical application

In the prototype, the OrchestratorAgent asks "are you a doctor, patient, or pharmacist?" and trusts the answer. In production, that trust must come from a verified identity token — not a chat message.

The good news: the agent code barely changes. The auth logic moves into your API layer (Phase 4), and the agents simply receive a pre-verified role. Here's the architecture:

👤
User logs in via Azure AD / Entra ID
Single sign-on with their hospital or university Microsoft account. Roles are assigned in Azure AD groups (e.g. "Medical-Doctors", "Medical-Patients").
🔐
API receives a JWT bearer token
Every request includes an Authorization header. Your FastAPI endpoint validates the token and extracts the user's role and ID from the claims.
🤖
Role is injected into the agent context
The verified role and patient_id are prepended to the user's message before it reaches the OrchestratorAgent. The LLM never decides who you are.

Install the Azure auth library

terminal
pip install azure-identity msal fastapi python-jose[cryptography] httpx

Token validation middleware

Create auth.py in your project root:

auth.py
from fastapi import HTTPException, Security
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from jose import jwt, JWTError
import httpx, os

TENANT_ID  = os.environ["AZURE_TENANT_ID"]
CLIENT_ID  = os.environ["AZURE_CLIENT_ID"]
JWKS_URL   = f"https://login.microsoftonline.com/{TENANT_ID}/discovery/v2.0/keys"

# Role mapping — Azure AD group names → internal role strings
ROLE_MAP = {
    "Medical-Doctors":    "doctor",
    "Medical-Patients":   "patient",
    "Medical-Pharmacists": "pharmacist",
}

security = HTTPBearer()

async def get_current_user(credentials: HTTPAuthorizationCredentials = Security(security)):
    token = credentials.credentials
    try:
        # Fetch signing keys from Azure AD
        async with httpx.AsyncClient() as client:
            keys = (await client.get(JWKS_URL)).json()

        payload = jwt.decode(token, keys, algorithms=["RS256"],
                               audience=CLIENT_ID)

        user_id = payload.get("oid")   # Azure object ID
        groups  = payload.get("groups", [])

        # Map Azure AD group to internal role
        role = None
        for group in groups:
            if group in ROLE_MAP:
                role = ROLE_MAP[group]
                break

        if not role:
            raise HTTPException(status_code=403, detail="No recognized medical role assigned")

        patient_id = payload.get("extension_patientId", None)  # custom AD attribute
        return {"user_id": user_id, "role": role, "patient_id": patient_id}

    except JWTError:
        raise HTTPException(status_code=401, detail="Invalid or expired token")
⚠️ Don't skip this step for real patient data
HIPAA and most institutional data governance policies require that access to patient records be governed by a verified identity system — not a self-reported role in a chat message. Even for a pilot or internal tool, use Azure AD from day one. Retrofitting auth later is much harder than building it in.
3
Switch to a Persistent Vector Store
Phase 2 cont. — Replace in-memory ChromaDB

The lab uses chromadb.Client() — in-memory, lost on restart. Production needs persistent storage. You have two good options:

OptionBest forSetup effortCost
ChromaDB server (self-hosted) Staying close to the lab code, on-prem or VM deployment Low — one Docker command Infrastructure only
Azure AI Search Full Azure integration, managed service, enterprise scale Medium — portal setup + SDK swap Per-query pricing

Option A — Persistent ChromaDB server

The smallest change from the lab code — just point to a server instead of in-memory:

terminal — run ChromaDB as a persistent server
# Run ChromaDB with data persisted to ./chroma-data
docker run -d \
  -p 8000:8000 \
  -v $(pwd)/chroma-data:/chroma/chroma \
  chromadb/chroma:latest

Then in vector_store.py, replace the client line:

vector_store.py — one-line change
# Replace this (in-memory — lab version):
chroma_client = chromadb.Client()

# With this (persistent server — production version):
chroma_client = chromadb.HttpClient(host="localhost", port=8000)
# In Azure, replace "localhost" with your container's internal hostname

Option B — Azure AI Search

Fully managed, integrates with Azure RBAC and Key Vault. Install the SDK and update the retrieval functions:

terminal
pip install azure-search-documents azure-identity
vector_store_azure.py
from azure.search.documents import SearchClient
from azure.search.documents.models import VectorizedQuery
from azure.identity import DefaultAzureCredential
from sentence_transformers import SentenceTransformer
import os

embedder = SentenceTransformer("all-MiniLM-L6-v2")

search_client = SearchClient(
    endpoint=os.environ["AZURE_SEARCH_ENDPOINT"],
    index_name="medical-records",
    credential=DefaultAzureCredential()  # uses Managed Identity — no keys!
)

def query_patient(question: str, patient_id: str) -> str:
    vector = VectorizedQuery(
        vector=embedder.encode(question).tolist(),
        k_nearest_neighbors=5,
        fields="embedding"
    )
    results = search_client.search(
        search_text=None,
        vector_queries=[vector],
        filter=f"patient_id eq '{patient_id}'",  # OData filter — same concept as ChromaDB where
        select=["content", "patient_id", "section"]
    )
    return "\n\n".join([
        f"[Patient {r['patient_id']} | {r['section'].upper()}] {r['content']}"
        for r in results
    ])
✅ Notice what didn't change
The filter parameter in Azure AI Search does the same job as ChromaDB's where clause — it enforces data boundaries at the retrieval layer. The agent code and tool signatures stay identical. Only the backend swaps.
4
Wrap the Team as a REST API with FastAPI
Phase 2 cont. — Any client can now call your agent system

FastAPI gives your agent system an HTTP interface so web apps, mobile apps, EHR systems, or any other client can send queries and receive responses. This is also where the verified role from Phase 2 gets injected into the agent context.

terminal
pip install fastapi uvicorn
main.py
from fastapi import FastAPI, Depends
from pydantic import BaseModel
from auth import get_current_user
from team import team, run
import asyncio, logging

app = FastAPI(title="Medical Records AI API", version="1.0")
logger = logging.getLogger(__name__)

class QueryRequest(BaseModel):
    message: str

class QueryResponse(BaseModel):
    response: str
    role_used: str
    patient_id: str | None

@app.post("/query", response_model=QueryResponse)
async def query(
    request: QueryRequest,
    user: dict = Depends(get_current_user)   # ← auth runs here, before agent
):
    role       = user["role"]
    patient_id = user.get("patient_id")
    user_id    = user["user_id"]

    # Inject verified role into message — LLM never decides who the user is
    enriched_message = (
        f"[SYSTEM: Verified user. Role={role}. PatientID={patient_id or 'N/A'}]\n"
        f"User query: {request.message}"
    )

    # Audit log every request
    logger.info(f"QUERY user={user_id} role={role} patient={patient_id} msg={request.message[:80]}")

    result = await run(enriched_message)

    return QueryResponse(
        response=result.messages[-1].content,
        role_used=role,
        patient_id=patient_id
    )

@app.get("/health")
async def health(): return {"status": "ok"}

Run it locally

terminal
uvicorn main:app --reload --port 8000

# FastAPI auto-generates interactive docs at:
# http://localhost:8000/docs

Test it with curl

terminal
curl -X POST http://localhost:8000/query \
  -H "Authorization: Bearer YOUR_TOKEN_HERE" \
  -H "Content-Type: application/json" \
  -d '{"message": "What was my diagnosis?"}'
🔑 What changed in the OrchestratorAgent
The OrchestratorAgent's system prompt no longer needs to ask for the user's role — it receives it in the enriched message. Update its prompt to: "You will receive a verified role and patient ID at the start of each message in [SYSTEM: ...] tags. Route directly to the appropriate specialist agent without asking for role confirmation." This makes the API feel seamless — no conversational role-check friction for the end user.

# Install system dependencies for Whisper
RUN apt-get update && apt-get install -y ffmpeg && rm -rf /var/lib/apt/lists/*

Build and test locally

terminal
# Build the image
docker build -t medical-demo:latest .

# Run it — pass secrets as environment variables, never bake them in
docker run -p 8000:8000 \
  -e AZURE_OPENAI_API_KEY=your_key \
  -e AZURE_OPENAI_ENDPOINT=your_endpoint \
  -e AZURE_DEPLOYMENT_NAME=gpt-4.1 \
  -e AZURE_API_VERSION=2024-12-01-preview \
  -e AZURE_TENANT_ID=your_tenant \
  -e AZURE_CLIENT_ID=your_client_id \
  medical-demo:latest
⚠️ Never bake secrets into the image
Do not copy your .env file into the Docker image and do not set secret values in the Dockerfile itself. Always pass them as environment variables at runtime, or better — use Azure Key Vault with Managed Identity (covered next) so secrets never appear in any config file at all.

Deploy to Azure
Phase 3 cont. — Three options depending on your scale
OptionBest forAutoscalesComplexity
Azure Container Apps Most AI API deployments — serverless, scales to zero Yes Low
Azure App Service Always-on apps, simpler pricing model Manual Low
Azure Kubernetes Service (AKS) High-throughput, multiple microservices, full control Yes High

For a medical records AI system at pilot or departmental scale, Azure Container Apps is the right choice — serverless, scales to zero when idle, and integrates cleanly with Managed Identity and Key Vault.

Deploy with Azure Container Apps

terminal — Azure CLI
# 1. Log in to Azure
az login

# 2. Create a container registry to store your image
az acr create --name medicaldemoacr --resource-group YOUR_RG \
  --sku Basic --admin-enabled true

# 3. Build and push your image to the registry
az acr build --registry medicaldemoacr \
  --image medical-demo:latest .

# 4. Create the Container Apps environment
az containerapp env create \
  --name medical-env \
  --resource-group YOUR_RG \
  --location eastus

# 5. Deploy the container app
az containerapp create \
  --name medical-demo \
  --resource-group YOUR_RG \
  --environment medical-env \
  --image medicaldemoacr.azurecr.io/medical-demo:latest \
  --target-port 8000 \
  --ingress external \
  --min-replicas 0 \
  --max-replicas 10 \
  --secrets "oai-key=YOUR_KEY" \
  --env-vars "AZURE_OPENAI_API_KEY=secretref:oai-key"

Enable Managed Identity (replace API keys with identity-based access)

Once deployed, enable a system-assigned Managed Identity so your container can access Azure OpenAI and Key Vault without any stored credentials:

terminal — Azure CLI (same on Mac and PowerShell)
# Enable Managed Identity on your container app
az containerapp identity assign \
  --name medical-demo \
  --resource-group YOUR_RG \
  --system-assigned

# Grant it access to your Azure OpenAI resource
az role assignment create \
  --assignee YOUR_MANAGED_IDENTITY_PRINCIPAL_ID \
  --role "Cognitive Services OpenAI User" \
  --scope /subscriptions/YOUR_SUB/resourceGroups/YOUR_RG/providers/Microsoft.CognitiveServices/accounts/YOUR_OAI_RESOURCE

Then update your model client in team.py to use Managed Identity instead of an API key:

team.py — swap API key for Managed Identity
from azure.identity import DefaultAzureCredential, get_bearer_token_provider

# DefaultAzureCredential picks up Managed Identity automatically in Azure
token_provider = get_bearer_token_provider(
    DefaultAzureCredential(),
    "https://cognitiveservices.azure.com/.default"
)

model_client = AzureOpenAIChatCompletionClient(
    azure_deployment=os.environ["AZURE_DEPLOYMENT_NAME"],
    api_version=os.environ["AZURE_API_VERSION"],
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    azure_ad_token_provider=token_provider,  # ← no api_key needed
)

✅ What the lab taught you that transfers directly
The agent architecture, system prompts, tool function signatures, and retrieval filtering logic from the lab are identical in production. None of that changes.
